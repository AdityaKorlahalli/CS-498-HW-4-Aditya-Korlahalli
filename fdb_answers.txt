FoundationDB decouples the TMS from the SS to allow each to scale independently according to workload needs. The TMS layer handles the heavy lifting of transaction ordering, conflict detection, and logging, while the SS layer focuses purely on storing data on disk and serving read requests.
The Resolver component is responsible for detecting read-write conflicts between transactions that are concurrent. This check occurs during the commit process when a transaction has been submitted to the Proxies, but before it is written to the Log Servers. The Resolver makes sure that the keys read by the transaction have not been modified by another committed transaction, since the current transaction's read version was assigned.
The purpose of the read version is to provide a transaction with a consistent picture of the database at any point in time. It makes sure that all reads within that transaction are isolated and see a globally consistent state, which is the foundation of FDB’s Strict Serializability. By assigning a read version from the Sequencer, FDB guarantees that the transaction will not see partial updates from other concurrent operations.
Snapshot reads differ from standard reads because they don’t add the keys being read to the transaction’s conflict set. This means that even if another transaction modifies those keys, the current transaction won’t fail because of a conflict. Snapshot reads are used when high concurrency is more important, such as calculating unimportant statistics or other background tasks.
FDB achieves cheap recovery because it uses LogServers to store a short history of mutations before they are sent to the StorageServers. During recovery, a new set of transaction nodes is gained, and they identify the last consistent version from the LogServers to continue operating. Since StorageServers are always pulling data from LogServers in the background, there is no need to replay a massive, centralized redo log before the system can become available again.
